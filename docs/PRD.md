## 1. 제품 요약
Coin Lab은 업비트(Upbit) 시장을 대상으로 하는 전략 실험실이자 실전 트레이딩 콘솔이다. 사용자는 전략을 버전 단위로 관리하고, 검증된 전략을 BACKTEST, PAPER, LIVE 모드로 실행하며, 실시간 모니터링 화면에서 세션 상태, 신호, 주문, 포지션, 리스크를 통합적으로 관찰할 수 있어야 한다.

제품의 핵심 가치는 다음 세 가지다.
- 연구와 실행을 분리된 도구가 아니라 하나의 운영 표면에서 연결한다.
- BACKTEST에서 검증한 전략을 PAPER와 LIVE로 안전하게 전이한다.
- 실시간 장애, 지연, 리스크 블록, 주문 상태를 즉시 확인하고 개입할 수 있게 한다.

## 2. 문제 정의
암호화폐 전략 운용자는 보통 다음 문제를 동시에 겪는다.
- 전략 정의와 버전 관리가 문서, 코드, 수기 기록으로 흩어져 있다.
- 백테스트 결과와 실시간 실행 결과가 같은 기준으로 연결되지 않는다.
- PAPER와 LIVE 전환 시 운영자 실수나 설정 누락으로 실제 주문 리스크가 발생한다.
- 실시간 모니터링에서 신호, 주문, 포지션, 리스크, 로그가 한 화면에서 이어지지 않는다.
- 재연결, 중복 이벤트, 지연 데이터 같은 실시간 시스템 특성이 운영 화면에 충분히 드러나지 않는다.

Coin Lab은 이 단절을 줄여 전략 정의, 검증, 실행, 관찰, 사후 분석의 흐름을 일관된 제품 경험으로 묶는 것을 목표로 한다.

## 3. 목표

### 3.1 제품 목표
- 전략을 버전 중심으로 등록, 검토, 검증, 실행할 수 있다.
- 검증된 전략을 즉시 BACKTEST, PAPER, LIVE 중 하나로 실행할 수 있다.
- 실시간 모니터링에서 세션별 상태와 리스크를 빠르게 파악하고 운영자가 개입할 수 있다.
- 시장 데이터 수집, 전략 평가, 실행, 로그가 같은 추적 맥락(`trace_id`) 안에서 이어진다.
- LIVE 모드는 명시적 확인과 환경 가드가 없으면 절대 시작되지 않는다.

### 3.2 비즈니스 목표
- 전략 아이디어에서 실행까지 걸리는 시간을 단축한다.
- 실거래 전 PAPER 검증 비중을 높여 운영 사고 가능성을 낮춘다.
- 운영자가 한 화면에서 여러 세션을 비교하며 의사결정할 수 있게 한다.

## 4. 비목표
- 다중 거래소 동시 지원은 MVP 범위에 포함하지 않는다.
- 모바일 앱 전용 UX는 MVP 범위에 포함하지 않는다.
- 포트폴리오 최적화, 자동 리밸런싱, 자산배분 엔진은 MVP 핵심 범위가 아니다.
- 완전 자동 전략 생성, 자동 튜닝, LLM 기반 전략 추천은 MVP 범위가 아니다.
- 부팅 시 PAPER/LIVE 세션 자동 시작은 허용하지 않는다.

## 5. 대상 사용자

### 5.1 1차 사용자
- 개인 또는 소규모 팀의 시스템 트레이더
- 전략을 직접 정의하고 운영까지 담당하는 개발자형 운영자
- PAPER/LIVE 전환 전에 전략 상태와 리스크를 검토해야 하는 의사결정자

### 5.2 사용자 니즈
- 전략 버전별 변경 이력과 검증 상태를 빠르게 확인하고 싶다.
- 실시간 세션이 실제로 어떤 종목을 감시하고 어떤 신호를 냈는지 알고 싶다.
- LIVE 세션에서 위험 신호가 발생하면 즉시 정지 또는 강제 종료하고 싶다.
- 백테스트 성과와 실시간 성과를 같은 제품 안에서 비교하고 싶다.

## 6. 제품 원칙과 제약
- 지원 시장 범위는 Upbit의 KRW, BTC, USDT 마켓이다.
- 실행 단위는 단일 전략 세션이다.
- 하나의 세션은 정확히 하나의 `strategy_version_id`에 귀속된다.
- 전략은 종목 독립형으로 평가한다.
- 전략 정의 방식은 JSON DSL + Python plugin 혼합 구조를 따른다.
- 실행 모드는 BACKTEST, PAPER, LIVE로 분리한다.
- 기본 부팅 상태에서는 전략이 보여도 세션은 자동 실행되지 않는다.
- 모니터링 화면은 최대 4개 세션까지 병렬 비교를 지원해야 한다.
- 실시간 파이프라인은 `raw -> normalized event -> snapshot -> evaluate -> execute` 흐름을 유지한다.
- LIVE 모드는 `confirm_live`, `acknowledge_risk`, 환경 플래그, 주문 테스트 가드가 모두 충족될 때만 시작할 수 있다.

## 7. 핵심 사용자 흐름

### 7.1 전략 등록과 검증
1. 사용자는 전략 목록에서 전략을 선택하거나 새 전략을 만든다.
2. 전략 버전을 생성하고 JSON DSL 또는 편집 UI를 통해 내용을 수정한다.
3. 사용자는 검증을 실행하고 에러/경고를 확인한다.
4. 검증된 최신 버전만 실행 후보가 된다.

### 7.2 백테스트 실행
1. 사용자는 전략 상세 또는 백테스트 화면에서 실행 조건을 선택한다.
2. 백테스트를 실행하면 run 이력이 생성되고 진행 상태와 결과를 조회할 수 있다.
3. 사용자는 성과 지표, 거래 목록, equity curve를 확인한다.

### 7.3 PAPER 세션 실행
1. 사용자는 Strategies 화면에서 검증된 최신 전략 버전을 선택한다.
2. 모드, 유니버스 소스, 최대 종목 수를 지정한다.
3. 세션 시작 성공 시 즉시 `RUNNING` 상태가 반환되고 Monitoring 화면으로 이동한다.

### 7.4 LIVE 세션 실행
1. 사용자는 LIVE 모드를 선택한다.
2. 두 개의 위험 확인 스위치를 켠다.
3. 시스템은 환경 플래그, API 키, 최소 주문 금액, 주문 테스트 가드를 확인한다.
4. 세션 시작 성공 시 LIVE 배지와 고정 경고가 즉시 표시된다.

### 7.5 실시간 운영과 개입
1. 운영자는 세션 상태, 차트, 신호, 주문, 포지션, 리스크 이벤트를 본다.
2. 필요한 경우 Stop 또는 Emergency Kill을 실행한다.
3. 운영자는 로그와 이벤트 이력을 통해 원인을 추적한다.

## 8. 기능 요구사항

### 8.1 Dashboard
- 시스템 전반 상태를 한 눈에 보여줘야 한다.
- `running_session_count`, `paper_session_count`, `live_session_count`, `degraded_session_count`, `active_symbol_count`를 표시해야 한다.
- 전략 카드, 후보군 요약, 리스크 경고, 최근 신호를 함께 제공해야 한다.
- 세션이 0개인 첫 부팅 상태도 정상 UX로 처리해야 한다.

### 8.2 Strategies
- 전략 목록은 이름, 최신 버전, 타입, 태그, 활성 여부, 최근 성과, 수정 시각을 보여줘야 한다.
- 전략 상세는 최신 버전 요약, 성과 카드, 버전 이력, 관련 세션/백테스트를 보여줘야 한다.
- 최신 버전의 canonical JSON snapshot을 확인할 수 있어야 한다.
- 검증되지 않은 최신 버전은 PAPER/LIVE 시작이 차단되어야 한다.

### 8.3 Strategy Edit
- 편집 UX는 Form + JSON 동기화 방식을 따라야 한다.
- Basic, Market & Universe, Entry, Reentry, Position, Exit, Risk, Execution, Backtest, Diff Preview, JSON Editor, Validation 탭을 제공해야 한다.
- 검증 결과는 즉시 표시되어야 한다.
- 버전 간 diff preview를 제공해야 한다.

### 8.4 Session Launch
- Strategies 화면에서 BACKTEST, PAPER, LIVE를 시작할 수 있어야 한다.
- 성공한 `POST /api/v1/sessions` 호출은 즉시 `RUNNING` 세션을 반환해야 한다.
- 세션 시작은 자동 부팅이 아니라 사용자 명시 동작이어야 한다.
- LIVE 시작은 추가 확인과 안전 가드가 없으면 실패해야 한다.

### 8.5 Monitoring
- 최대 4개 세션을 병렬 비교할 수 있어야 한다.
- 좌측 패널에서 세션 선택, 비교 대상, active universe를 확인할 수 있어야 한다.
- 중앙 패널에서 심볼/타임프레임 기준 차트와 오버레이를 볼 수 있어야 한다.
- 우측 `Session Detail` 탭 패널에서 Event Log, Strategy Explain, Signals, Orders, Risk를 지원해야 한다.
- `Signals` 탭에서 선택한 신호와 차트 marker 선택은 같은 `Strategy Explain` 상태를 공유해야 한다.
- LIVE 세션은 PAPER와 명확히 다른 색상 체계와 경고 배너를 가져야 한다.
- Stop과 Emergency Kill은 항상 접근 가능해야 한다.

### 8.6 Backtests
- 새 백테스트 실행, 히스토리 조회, 결과 상세 확인을 지원해야 한다.
- 성과 지표에는 최소 총수익률, MDD, 승률, Profit Factor, 거래 수가 포함되어야 한다.
- equity curve와 최근 거래 목록을 시각적으로 확인할 수 있어야 한다.

### 8.7 Compare
- Monitoring 워크스페이스에서는 세션 단위 병렬 비교를 지원해야 한다.
- 전용 Compare 화면은 MVP에서 백테스트 run 비교를 우선 지원할 수 있다.
- 최소 비교 지표는 총수익률, MDD, 승률, Profit Factor, 거래 수를 포함해야 한다.
- 세션 비교와 백테스트 비교의 용어/범위는 추후 문서 정합화가 필요하다.

### 8.8 Logs
- System, Strategy Execution, Order Simulation, Risk Control 채널을 지원해야 한다.
- 로그는 세션 단위 또는 전역 단위로 필터링 가능해야 한다.
- 각 로그 항목은 최소 시간, 채널, 세션, 이벤트 타입, 메시지, payload를 포함해야 한다.

### 8.9 Settings
- Upbit 연결 설정, 저장소 설정, 유니버스 정책, 차트 설정, 리스크 기본값, LIVE 보호 설정을 노출해야 한다.
- MVP에서는 일부 항목이 읽기 전용 상태 설명 카드로 시작할 수 있다.
- 장기적으로는 제품 내 설정 변경과 저장이 가능해야 한다.

## 9. 데이터 및 시스템 요구사항
- 백엔드는 FastAPI 기반 REST + WebSocket 인터페이스를 제공해야 한다.
- 프론트엔드는 React + TypeScript 기반으로 React Query와 Zustand 책임 분리를 유지해야 한다.
- 시장 데이터는 업비트 WebSocket에서 수집되고 내부 정규화 이벤트로 변환되어야 한다.
- 중복 이벤트 제거, 순서 뒤바뀜 대응, 재연결, 백프레셔 제어가 필요하다.
- chart-ready candle, session health/performance, universe 상태는 API 친화적 형태로 materialized state를 가질 수 있어야 한다.
- 저장소는 기본 메모리 모드를 지원하고, Postgres/Supabase 기반 지속 저장을 선택적으로 지원해야 한다.

## 10. 비기능 요구사항

### 10.1 안정성
- 재연결 상태, 늦은 이벤트 수, snapshot consistency 상태가 운영 화면에 노출되어야 한다.
- 세션, 주문, 포지션, 신호는 상태 문자열을 API/UI/테스트 전반에서 동일하게 사용해야 한다.

### 10.2 성능
- 대시보드와 핵심 목록 화면은 일반적인 로컬 개발 환경에서 수 초 내 첫 데이터가 보여야 한다.
- 차트 스트림은 배치 업데이트를 사용해야 한다.
- 로그와 대규모 리스트는 가상화 또는 행 제한 전략을 적용해야 한다.

### 10.3 안전성
- LIVE 주문은 명시적 사용자 동작과 환경 설정 없이는 절대 발동되면 안 된다.
- Emergency Kill은 LIVE 세션에서 항상 시인성 높은 위치에 있어야 한다.
- 모드 혼동을 막기 위해 PAPER와 LIVE의 시각 체계는 명확히 구분되어야 한다.

### 10.4 감사 가능성
- 모든 핵심 API 응답은 `trace_id`와 `timestamp`를 포함해야 한다.
- 로그는 최소 `timestamp`, `trace_id`, `mode`, `strategy_version_id`, `session_id`, `symbol`, `event_type`를 담아야 한다.

### 10.5 문서 일관성
- API, 상태값, UI 용어, 전략 JSON shape는 관련 SSOT 문서와 함께 유지되어야 한다.
- 동작 변경 시 관련 명세와 테스트가 함께 갱신되어야 한다.

## 11. MVP 범위

### 포함
- 전략 목록/상세/버전/검증
- BACKTEST 실행 및 결과 조회
- PAPER/LIVE 세션 시작, 정지, 강제 종료
- 실시간 대시보드 및 모니터링
- 후보군 조회, 세션 하위 리소스 조회
- 로그 채널 조회
- 업비트 WebSocket 기반 실시간 데이터 수집

### 제외 또는 후순위
- 멀티 거래소
- 멀티 계정/권한 체계
- 모바일 전용 앱
- 자동 전략 생성/튜닝
- 포트폴리오 차원의 리밸런싱 엔진

## 12. 성공 지표
- 검증된 전략 기준 세션 시작 요청의 대부분이 단일 재시도 없이 `RUNNING`으로 전이되어야 한다.
- LIVE 시작 차단 조건 누락 시 100% 차단되어야 한다.
- 운영자는 모니터링 화면에서 세션 상태, 신호, 포지션, 주문, 리스크를 같은 컨텍스트에서 확인할 수 있어야 한다.
- 백테스트 결과 화면에서 핵심 성과 지표와 거래 이력을 바로 해석할 수 있어야 한다.
- 첫 부팅 상태에서 세션이 0개여도 사용자가 “비정상”으로 오해하지 않도록 안내되어야 한다.

## 13. 현재 구현 기준 갭 메모
- Strategies 화면은 현재 목록, 상세, 버전 이력, 세션/백테스트 실행 흐름은 제공하지만, 정식 편집 폼과 diff preview는 아직 요구사항 수준에 가깝다.
- Monitoring 화면은 세션 선택, 차트, `Session Detail` 5탭, 리스크/로그 inspection 흐름을 제공하지만, 4세션 병렬 비교 UX는 추가 확장이 필요하다.
- Compare 화면은 현재 백테스트 run 비교에 초점이 맞춰져 있으며, 문서 전반의 세션 비교 정의와 완전 일치하지 않는다.
- Settings 화면은 현재 설명형 상태 카드 중심이며, 실제 저장 가능한 설정 패널은 후속 구현이 필요하다.
- 기본 저장소는 메모리이며, 재시작 시 세션 이력이 유지되지 않는다.

## 14. 오픈 이슈
- 전용 Compare 화면의 1차 비교 단위를 세션으로 볼지, 백테스트 run으로 볼지 명확한 제품 결정이 필요하다.
- 전략 편집 화면의 1차 구현 범위를 “JSON 중심”으로 시작할지 “폼 중심”으로 시작할지 우선순위 결정이 필요하다.
- Supabase/Postgres를 운영 기본 저장소로 승격할 시점과 조건을 정해야 한다.
- 사용자 인증, 권한, 감사 추적 범위를 언제 제품 범위에 포함할지 결정이 필요하다.

## 15. 출시 판단 기준
- 적어도 하나 이상의 검증된 전략을 BACKTEST, PAPER, LIVE 흐름으로 연결할 수 있어야 한다.
- Monitoring에서 LIVE/PAPER 상태와 핵심 리스크를 구분해 운영자가 즉시 개입할 수 있어야 한다.
- REST 및 WebSocket 계약이 프론트와 백엔드 사이에서 안정적으로 동작해야 한다.
- 첫 실행 가이드와 실제 제품 동작이 어긋나지 않아야 한다.
