## 1. 문서 목적

이 문서는 AI가 UI를 구현할 때 시각적 품질을 일정 수준 이상으로 유지하도록 만드는 디자인 규칙 문서다.
목표는 화면을 밋밋하게 만드는 것이 아니라, `DESIGN_SYSTEM.md`에 정의된 다크 프리미엄 트레이딩 UI를 정돈되고 일관되게 구현하게 하는 것이다.

이 문서는 특히 다음 문제를 방지한다.

- 기능만 나열된 투박한 화면
- 페이지마다 다른 스타일과 밀도
- 여백 부족으로 답답한 구성
- 모든 요소가 동시에 강조되는 혼란스러운 화면
- 테이블, 폼, 카드, 버튼의 일관성 붕괴

---

## 2. 디자인 철학

이 프로젝트에서 좋은 디자인은 아래 기준을 만족해야 한다.

- 정보 계층이 명확하다
- 여백과 정렬이 안정적이다
- 과한 장식 없이 신뢰감이 있다
- 중요한 정보와 중요한 액션이 먼저 보인다
- 같은 역할의 요소가 같은 방식으로 보인다
- 복잡한 기능도 차분하고 정리된 화면처럼 느껴진다
- 색상보다 구조와 간격으로 품질을 만들되, 필요한 경우 절제된 glow와 accent를 사용한다

디자인의 우선순위는 다음과 같다.

1. 사용성
2. 정보 구조의 명확성
3. 일관성
4. 시각적 완성도
5. 장식성

## 3. 톤앤매너

기본 톤은 다음을 따른다.

- Ultra dark base 위의 premium crypto dashboard 스타일
- 프라이머리 블루/시안 accent를 제한적으로 쓰는 고밀도 금융 UI
- 모던하고 정돈된 운영 콘솔
- 화려함보다 신뢰감 우선, 그러나 지나치게 밋밋하지 않음
- 장식보다 정보 전달 우선
- 강한 개성보다 안정적인 사용성 우선
- 절제된 디테일 > 범용 SaaS 느낌
- 전문성 > 장난스러움

### 금지 방향

- `DESIGN_SYSTEM.md` 밖의 임의 색상 체계 사용
- 형광 포인트를 화면 전체에 남발하는 구성
- 무거운 그라데이션, 무거운 그림자, 과한 테두리
- 요소마다 개성이 너무 강한 스타일
- “예뻐 보이기” 위해 구조를 희생하는 디자인

## 4. Spacing Rules

기본 spacing unit은 `DESIGN_SYSTEM.md`의 spacing token(`space-4` ~ `space-40`)을 사용한다.

### 기본 규칙

- 최소 단위: 4px
- 기본 단위: 8px
- 작은 간격: 8px
- 기본 요소 간격: 12~16px
- 카드 내부 padding: 16~24px
- 섹션 간 간격: 24~32px
- 큰 섹션 간 간격: 32~48px
- 화면 좌우 여백은 충분히 확보한다

### 적용 규칙

- label과 input 사이: 6~8px
- input과 input 사이: 12~16px
- 필터 영역과 테이블 사이: 16~24px
- 카드와 카드 사이: 16~24px
- 타이틀과 설명 사이: 4~8px

---

### 금지사항
- spacing token을 무시하고 임의의 간격 값을 남발하지 않는다
- 카드 padding과 섹션 간격을 화면마다 다르게 사용하지 않는다

---

## 5. Layout Rules

모든 화면은 “기능 나열”이 아니라 “정보 구조”를 기준으로 설계한다.

### 기본 화면 구조

가능한 한 아래 구조를 따른다.

- Page Header
- Summary / Description
- Filters / Controls
- Main Content
- Secondary Actions / Supporting Info

### 레이아웃 원칙

- 한 화면에 너무 많은 박스와 구획을 만들지 않는다
- 한 행에 너무 많은 정보 블록을 나열하지 않는다
- 스크롤 구간이 여러 개 생기는 구조는 최소화한다
- 중요한 영역은 화면 상단에 배치한다
- 넓은 화면에서는 좌우 균형과 최대 너비를 고려한다

### 관리자 화면 원칙

- 목록/필터/상태/액션을 예측 가능한 위치에 둔다
- 복잡한 정보는 섹션 또는 카드로 묶되, 과도한 카드 분할은 피한다
- 작업 중심 화면에서는 핵심 액션을 명확히 드러낸다

---

## 6. Visual Hierarchy Rules

시각적 계층 구조는 강한 색상이 아니라 위치, 크기, 굵기, 여백으로 만든다.

### 계층 단계

- Level 1: 페이지 제목 / 핵심 KPI / 가장 중요한 액션
- Level 2: 섹션 제목 / 서브 핵심 정보
- Level 3: 일반 본문 / 테이블 본문 / 폼 필드
- Level 4: 보조 설명 / 도움말 / 캡션

### 원칙

- 한 화면에서 가장 중요한 포인트는 1~2개만 둔다
- 모든 버튼과 정보가 동시에 강조되면 안 된다
- primary action은 한 영역에 1개만 강하게 둔다
- 보조 액션은 secondary / text / ghost 스타일을 사용한다
- 같은 레벨의 정보는 같은 정렬, 같은 여백, 같은 시각 무게를 가진다

---

## 7. Typography Rules

폰트 크기와 굵기는 적은 단계로 일관되게 사용한다.

### 권장 체계

- 페이지 제목: `title.lg` 또는 `display.md`
- 섹션 제목: `title.md`
- 카드 제목: `title.sm`
- 본문: `body.md` 또는 `body.lg`
- 보조 텍스트: `body.sm` 또는 `caption.lg`
- 캡션/메타 정보: `caption.sm`, `micro`

### 원칙

- 너무 많은 폰트 사이즈 단계를 만들지 않는다
- 굵기 차이도 제한적으로 사용한다
- 긴 설명은 시각적 우선순위를 낮춘다
- 작은 회색 글씨를 과도하게 남용하지 않는다
- 숫자/KPI는 주변 텍스트보다 확실히 도드라져야 한다
- 숫자, 가격, 수익률은 tabular numeric 사용을 우선한다

### 금지사항

- 한 화면에 5개 이상의 텍스트 계층을 만들지 않는다
- 텍스트 크기 차이만으로 모든 계층을 해결하려 하지 않는다
- 과하게 작은 텍스트로 정보를 숨기지 않는다

---

## 8. Color Usage Rules

### 기본 원칙

- 브랜드 컬러는 `DESIGN_SYSTEM.md`의 `brand.primary` 계열만 사용한다
- 배경은 `bg.canvas`, `bg.surface-*` 같은 중립 계열 위주로 유지한다
- 성공/경고/오류 색상은 semantic token(`success`, `warning`, `danger`, `info`) 용도로만 사용한다
- 텍스트는 대비가 충분해야 한다
- 색만으로 상태를 구분하지 않는다

### 권장 사용 방식

- 페이지 전체에서 강한 포인트 컬러는 브랜드 프라이머리 1개와 semantic color 정도로 제한한다
- 중립 배경 + 제한된 포인트 컬러 조합
- 배지/태그/상태칩은 공통 의미 체계를 따른다
- 브랜드 프라이머리는 전체 화면 면적의 소수 영역에만 남긴다
- 대시보드 `Performance History`의 전략 카드와 주요 전략 요약은 `brand.primary` 계열 안에서만 변주한다

### 금지사항

- 예뻐 보이게 하려고 다양한 포인트 색을 섞지 않는다
- gradient와 glow를 기본 해결책처럼 사용하지 않는다
- 빨강/초록만으로 상태를 전달하지 않는다
- 모든 카드나 박스에 배경색을 다르게 주지 않는다

---

## 9. Surface / Border / Shadow Rules

표면(surface) 스타일은 절제해서 사용한다.

### 원칙

- 카드, 패널, 섹션은 일관된 radius를 사용한다
- 그림자는 약하고 적게 사용한다
- border는 구획 분리 용도로만 사용한다
- surface 간 위계는 배경 톤 차이, border, 여백으로 구분한다

### 권장 값

- radius: `radius.sm` / `radius.md` / `radius.lg` / `radius.xl` 중 제한적으로 사용
- shadow: `glow.panel`, `shadow.sm`, `shadow.md` 수준만 사용
- border: `border.default` 또는 `border.soft`

### 금지사항

- 모든 요소에 진한 shadow를 넣지 않는다
- border와 shadow와 배경색을 동시에 과하게 쓰지 않는다
- radius 스타일이 화면마다 달라지지 않게 한다

---

## 10. Card Rules

카드는 정보를 그룹화할 때만 사용한다. 박스를 많이 만드는 것이 고급스러움을 의미하지 않는다.

### 원칙

- 카드 내부 정렬은 단순하고 좌측 정렬 중심으로 간다
- 카드 padding은 충분히 확보한다
- 카드 안에 너무 많은 강조 요소를 넣지 않는다
- 카드마다 다른 스타일을 주지 않는다
- 카드 헤더, 본문, 액션 구역을 명확히 나눈다

### 카드가 적합한 경우

- 서로 다른 정보 블록을 구분할 때
- KPI 또는 요약 정보를 묶을 때
- 설정 섹션이 명확히 나뉠 때

### 카드가 부적합한 경우

- 단순 목록 한 줄 정보까지 전부 카드화하는 경우
- 좁은 화면에 카드가 과도하게 중첩되는 경우

---

## 11. Table Rules

테이블은 정보 밀도가 높아지기 쉬우므로 특히 정돈이 중요하다.

### 구조 원칙

- 필터 영역 / 상단 액션 / 테이블 / 페이지네이션 영역을 명확히 분리한다
- 상태값은 텍스트만 두지 말고 공통 배지/태그 패턴을 사용한다
- 정렬 가능한 컬럼과 아닌 컬럼을 구분한다
- 긴 텍스트는 잘림, 툴팁, 상세 이동 규칙을 정한다

### 시각 원칙

- 컬럼 수를 무리하게 늘리지 않는다
- row height는 너무 촘촘하지 않게 유지한다
- 텍스트/숫자/날짜의 정렬 기준을 통일한다
- 빈 값 표시는 placeholder 규칙을 따른다

### 금지사항

- 액션 버튼을 행마다 과도하게 늘어놓지 않는다
- 모든 컬럼을 한 화면에 억지로 다 넣지 않는다
- 테이블 상단과 주변 여백을 생략하지 않는다

---

## 12. Form Rules

폼은 입력 편의성과 에러 이해가 우선이다.

### 구조 원칙

- 필드는 논리적인 그룹으로 나눈다
- 한 줄에 너무 많은 input을 넣지 않는다
- 필수값, 도움말, 에러 메시지 위치를 통일한다
- 긴 폼은 섹션 단위로 분리한다

### 시각 원칙

- label과 input 간 관계가 명확해야 한다
- 도움말은 본문보다 약하게, 에러는 충분히 인지 가능하게 표현한다
- 저장/취소 버튼은 예측 가능한 위치에 둔다

### 금지사항

- 폼 화면에 설명, 배너, 액션을 과도하게 섞지 않는다
- 에러 메시지를 필드와 멀리 떨어진 곳에 두지 않는다
- placeholder를 label 대용으로 사용하지 않는다

---

## 13. Button Rules

버튼은 우선순위 체계를 가져야 한다.

### 원칙

- 한 영역에서 primary button은 1개만 사용한다
- secondary, text, ghost, danger 버튼 역할을 명확히 구분한다
- destructive action은 명확히 구분한다
- 버튼 크기와 높이는 프로젝트 전체에서 통일한다

### 금지사항

- 같은 화면에 primary 스타일 버튼을 여러 개 두지 않는다
- 버튼마다 다른 radius, padding, 높이를 사용하지 않는다
- 중요하지 않은 액션까지 과하게 강조하지 않는다

---

## 14. Empty / Loading / Error Visual Rules

시각 상태도 디자인 품질의 일부다.

### Empty

- 오류처럼 보이지 않게 한다
- 왜 비어 있는지와 다음 행동을 함께 제시한다
- 일러스트/아이콘은 장식보다 정보 보조 수준으로만 사용한다

### Loading

- skeleton과 spinner 사용 기준을 구분한다
- 전체 화면 loading과 일부 영역 loading을 혼동하지 않는다
- 기존 데이터를 유지하는 경우 loading 느낌을 과도하게 주지 않는다

### Error

- 문제 상황임을 명확히 알리되 과하게 공격적이지 않게 표현한다
- 재시도 가능하면 명시적으로 제공한다
- 기술적인 문장을 그대로 노출하지 않는다

---

## 15. Consistency Rules

일관성은 미감보다 우선한다.

### 원칙

- 동일 기능은 동일 컴포넌트와 동일 스타일을 사용한다
- 동일 상태는 동일 배지/라벨/색상 규칙을 따른다
- 버튼, 입력 필드, 테이블, 카드, 모달은 공통 패턴을 재사용한다
- 페이지가 달라도 spacing, radius, typography 리듬은 유지한다

### 구현 지침

- 새 컴포넌트를 만들기 전에 기존 컴포넌트를 먼저 탐색한다
- 공통 패턴이 있으면 변형보다 재사용을 우선한다
- 예외 디자인은 반드시 근거가 있을 때만 허용한다

---

## 16. 반응형/리사이즈 규칙

AI가 데스크톱 기준으로만 촘촘하게 짜지 않도록 규칙을 둔다.

### 원칙

- 작은 화면에서 정보 블록은 자연스럽게 세로로 흐르도록 설계한다
- 좁아질 때 우선순위가 낮은 정보부터 접거나 아래로 내린다
- 필터/액션/요약 KPI가 좁은 화면에서도 깨지지 않게 한다
- 반응형에서도 핵심 액션 위치는 예측 가능해야 한다

---

## 17. 금지사항

- 색상으로만 예쁨을 해결하려 하지 말 것
- 모든 버튼을 강조 스타일로 만들지 말 것
- 한 화면에 너무 많은 카드와 박스를 쌓지 말 것
- 간격이 부족한 조밀한 UI를 만들지 말 것
- 같은 역할의 요소를 페이지마다 다른 스타일로 만들지 말 것
- 장식용 gradient, 과한 shadow, 과한 border를 남발하지 말 것
- 텍스트 크기 차이만으로 계층을 과하게 만들지 말 것
- 한 화면에 너무 많은 강조 포인트를 두지 말 것
- 관리자 UI를 마케팅 랜딩페이지처럼 꾸미지 말 것

DESIGN_SYSTEM.md를 시각 기준 문서로 사용해줘.

한 화면에서 가장 중요한 액션은 1개만 강하게 강조해줘.
같은 역할의 요소는 같은 패턴으로 반복해줘.
